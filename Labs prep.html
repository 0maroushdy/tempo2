TCP & UDP Socket Programming

Architecture: 3 files, 2 protocols

You built tcp_server.py, udp_server.py, and client.py. Both servers share the same process_message() logic — the first character is a command (A = sort descending, C = sort ascending, D = uppercase), and the rest is data. The client takes a CLI argument (tcp or udp) to choose which mode to run.

TCP Server — Key Design Choices

You used socket.AF_INET + socket.SOCK_STREAM for TCP. The server calls bind(), listen(), then loops on accept() — each accepted connection spawns a daemon thread via threading.Thread. This means the server handles multiple concurrent clients. Each thread runs handle_client() which loops on recv(1024) until the client disconnects (empty data or ConnectionResetError).

You also set SO_REUSEADDR — this prevents "Address already in use" errors when restarting the server quickly (allows reusing a port in TIME_WAIT state).

UDP Server — Key Design Choices

Used socket.SOCK_DGRAM. No listen(), no accept(), no threads. Just bind() then loop on recvfrom(1024). Each datagram is self-contained — the server processes it and sends the response back to the client's address using sendto(). No connection state is maintained.

Client — Unified Interface

TCP mode: connect() → loop of sendall() / recv() → persistent connection until user types "quit".
UDP mode: no connect() — just sendto() / recvfrom() each message independently.

The Socket API (Berkeley Sockets)

A socket is a software abstraction for a network endpoint, identified by (IP address, port number, protocol). It's the interface between the application layer and the transport layer. Python's socket module wraps the POSIX C socket API.

AF_INET and Port Numbers

AF_INET = IPv4 address family. Port numbers (0–65535) identify processes: well-known (0–1023), registered (1024–49151), dynamic/ephemeral (49152–65535). Your servers used 65432/65433 (dynamic range). The OS assigns an ephemeral port to the client automatically.

Threading and Concurrency

Your TCP server uses threads so multiple clients can connect simultaneously. Each accept() returns a new socket dedicated to that client. Without threads, the server would block on one client's recv() and couldn't handle others. The daemon=True flag means threads die when the main thread exits.

UDP doesn't need threads for this lab because it's message-oriented — recvfrom() handles any sender's datagram, processes it, and responds immediately. There's no persistent per-client state.

Expected Discussion Questions

Q&A ▼
Q1 What happens under the hood when the TCP client calls connect()? ▼
The OS initiates a 3-way handshake: (1) Client sends a SYN segment with a random initial sequence number. (2) Server's accept() unblocks, and the kernel replies with SYN-ACK containing its own ISN and ack = client ISN + 1. (3) Client sends ACK with ack = server ISN + 1. After this, both sides enter the ESTABLISHED state. In our code, connect() blocks until this completes.
Q2 Why does accept() return a new socket? Why not reuse the listening socket? ▼
The listening socket's job is to wait for new connections — it identifies the server's (IP, port). Each accepted connection needs its own socket to track that specific client's state (sequence numbers, buffer, etc.). TCP uses a 4-tuple (src IP, src port, dst IP, dst port) for demultiplexing — two different clients connecting to the same server port get separate sockets because their source IP/port differ.
Q3 What is SO_REUSEADDR and why did you use it? ▼
When a TCP connection closes, the port enters a TIME_WAIT state for ~2 minutes (to handle delayed packets). Without SO_REUSEADDR, trying to restart the server during TIME_WAIT gives "Address already in use" error. This option tells the OS to allow binding to a port that's still in TIME_WAIT.
Q4 Can UDP guarantee that the message arrives? What if it's lost? ▼
No. UDP provides no delivery guarantee. If a datagram is lost in the network (router drops it due to congestion, bit error, etc.), neither the sender nor receiver is notified. The client's recvfrom() would block forever waiting for a response that never comes. To handle this, you'd need to add application-level retransmission and timeouts — which is exactly what Lab 4 (RUDP) does.
Q5 What's the difference between send()/recv() and sendto()/recvfrom()? ▼
send()/recv() work on connected sockets (TCP, or UDP after connect()) — the destination is already known. sendto()/recvfrom() specify the destination address with each call — required for connectionless UDP where you might send to different addresses. recvfrom() also returns the sender's address, which the UDP server needs to send the reply back.
Q6 What does recv(1024) actually do? Is it guaranteed to return 1024 bytes? ▼
No! recv(1024) reads up to 1024 bytes from the kernel's receive buffer. TCP is a byte stream — data might arrive in chunks of any size. You could send 100 bytes in one send() and the receiver might get them across two recv() calls (e.g., 60 + 40). For guaranteed delivery of complete messages, you need a framing protocol (e.g., length prefix, delimiter). Our lab's simple messages fit in one segment, so this wasn't an issue.
Q7 Why did you use threads for TCP but not for UDP? ▼
TCP is connection-oriented — each client has a persistent connection with state, and recv() blocks. Without threads, handling one client would block all others. UDP is stateless and message-oriented — each recvfrom() gets a complete datagram from any sender, processes it immediately, and responds. No blocking on a specific client, so one loop handles all clients sequentially.
Q8 What is multiplexing and demultiplexing at the transport layer? ▼
Multiplexing = multiple application processes share one IP address, differentiated by port numbers. Outgoing data from different sockets is tagged with source/destination ports and sent through the same network interface. Demultiplexing = the transport layer routes incoming segments to the correct socket. UDP uses (dst IP, dst port) — all datagrams to that port go to the same socket. TCP uses the full 4-tuple (src IP, src port, dst IP, dst port) — different clients get routed to different sockets even on the same server port.
⚡
Pro tip for the discussion: If asked "what protocol would you choose for X?" — always justify with trade-offs. For a chat app: TCP for message delivery guarantees. For a live video call: UDP for low latency (dropped frames are better than delayed ones). For DNS: UDP for speed (short queries), with TCP fallback for large responses.

Core Concepts to Know Cold

Theory ▼

SMTP (Simple Mail Transfer Protocol)

An application-layer push protocol (RFC 5321) used to send emails. The client pushes the message to the server. Operates on port 25 (unencrypted), 587 (submission with STARTTLS), or 465 (implicit SSL). Key SMTP commands: EHLO (greeting), AUTH (authentication), MAIL FROM, RCPT TO, DATA, QUIT. The . on a line by itself terminates the message body.

STARTTLS vs implicit SSL: STARTTLS starts unencrypted on port 587, then upgrades to TLS mid-session. Port 465 starts encrypted from the first byte. STARTTLS is the recommended approach.

IMAP (Internet Message Access Protocol)

An application-layer pull protocol (RFC 3501) used to retrieve emails while keeping them on the server. IMAP supports folders, flags (read/unread), search, and partial fetches. Port 993 (SSL) or 143 (STARTTLS). Unlike POP3 which downloads and deletes, IMAP is designed for accessing mail from multiple devices.

MIME (Multipurpose Internet Mail Extensions)

Extends email beyond plain ASCII text. Adds support for attachments, HTML bodies, character encodings (UTF-8), and multipart messages. A multipart message has boundaries separating each part (text, HTML, attachments). Your code uses MIMEMultipart + MIMEText from Python's email library.

TLS/SSL — Transport Layer Security

Provides encryption, authentication, and integrity for the communication channel. TLS sits between the application layer and transport layer. The TLS handshake involves: (1) Client Hello with supported cipher suites, (2) Server Hello with chosen suite + certificate, (3) Key exchange (typically Diffie-Hellman), (4) Encrypted communication begins. Gmail requires TLS for both SMTP and IMAP.

Email Delivery Path

Sender's MUA (your client) → Sender's MSA (smtp.gmail.com) → MTA relay(s) → Recipient's MDA → Recipient's mailbox → Recipient's MUA (via IMAP/POP3). MUA = Mail User Agent (email client). MSA = Mail Submission Agent. MTA = Mail Transfer Agent (server-to-server relay). MDA = Mail Delivery Agent.

Q&A ▼
Q1 Walk me through the complete SMTP handshake in your code. ▼
(1) Open TCP connection to smtp.gmail.com:587. Server responds with 220 (ready). (2) Send EHLO — we identify ourselves and the server lists its capabilities (8BITMIME, SIZE, STARTTLS, etc.). (3) Send STARTTLS — server responds 220 Go ahead, then we perform TLS handshake to encrypt the channel. (4) Send EHLO again — required because the connection is now different (encrypted). (5) AUTH LOGIN with base64-encoded credentials. (6) MAIL FROM:<sender@gmail.com>. (7) RCPT TO:<recipient@example.com>. (8) DATA — followed by the MIME message — ended by \r\n.\r\n. (9) QUIT.
Q2 Why do you call ehlo() twice — once before and once after STARTTLS? ▼
The first EHLO establishes the initial handshake and lets us discover the server supports STARTTLS. After STARTTLS upgrades the connection to TLS, the session state is reset — the server may advertise different capabilities over the encrypted channel (e.g., AUTH is typically only listed after TLS). RFC 3207 requires a new EHLO after STARTTLS.
Q3 What's the difference between SMTP and IMAP? When is each used? ▼
SMTP is a push protocol — used to send emails from client to server and between servers. IMAP is a pull protocol — used to retrieve emails from a mailbox server. SMTP handles the outgoing direction; IMAP handles the incoming direction. When you compose and send, your client uses SMTP. When you check your inbox, your client uses IMAP. They serve complementary roles and use different ports and command sets.
Q4 What is MIME and why is it needed? ▼
SMTP was originally designed for 7-bit ASCII text only. MIME extends email to support non-ASCII characters (UTF-8), attachments (images, PDFs), HTML formatting, and multipart messages (text + HTML + attachments in one email). In code, MIMEMultipart creates a container, and MIMEText(body, "plain") creates a text part attached to it. The Content-Type header tells the receiver how to interpret each part.
Q5 Why does your GUI use threads for sending and receiving? ▼
Network operations (connecting to SMTP/IMAP servers, sending data) take time and can block. Tkinter runs on a single main thread — if we do a network call on the main thread, the entire GUI freezes (buttons become unresponsive, window can't be moved or closed) until the operation completes. By spawning a daemon thread for each network operation, the main thread stays free to handle GUI events. Results are passed back to the main thread using root.after(0, callback) which is thread-safe.
Q6 What is STARTTLS and how does it differ from using port 465? ▼
STARTTLS (port 587): The connection starts unencrypted in plain text. After the initial EHLO, the client sends STARTTLS to upgrade the existing connection to TLS. This is called an opportunistic upgrade. Implicit TLS (port 465): The connection is encrypted from the very first byte — the TLS handshake happens immediately upon TCP connection, before any SMTP commands. Port 587 + STARTTLS is the current recommended standard (RFC 8314 reintroduced 465 but 587 remains dominant).
Q7 What is an App Password and why can't you use your regular Gmail password? ▼
Gmail with 2FA enabled doesn't allow direct login via SMTP/IMAP with your regular password. App Passwords are 16-character codes generated in your Google account that give specific applications access without exposing your main password or requiring 2FA for each login. This is a security measure — if the app password is compromised, you can revoke just that password without changing your main credentials.
Q8 Compare IMAP vs POP3. ▼
IMAP keeps emails on the server, supports folders, flags, search, and partial fetches — ideal for multi-device access. POP3 downloads emails to the client and typically deletes them from the server (download-and-delete model). POP3 is simpler but doesn't sync across devices. IMAP is the modern standard for email retrieval.

Core Concepts to Know Cold

Theory ▼

TCP Sequence & Acknowledgment Numbers

Sequence Number: Identifies the first byte of data in this segment within the byte stream. Starts with a random ISN (Initial Sequence Number) chosen during the handshake. Each subsequent segment's seq = previous seq + data bytes sent.

Acknowledgment Number: "I've received everything up to byte X − 1, please send me byte X next." It's cumulative — acknowledges all bytes up to that number. Ack = Seq of received segment + length of data received.

SYN consumes 1 sequence number even though it carries no data. That's why the SYN-ACK's ack number = client ISN + 1.

Flow Control vs Congestion Control

Flow Control: Prevents the sender from overwhelming the receiver's buffer. Controlled by the receiver window (rwnd) field in each ACK. The receiver advertises how much buffer space it has available.

Congestion Control: Prevents the sender from overwhelming the network. Controlled by the congestion window (cwnd) maintained by the sender. The sender sends at the rate of min(cwnd, rwnd).

RTT Estimation

EstimatedRTT = (1 − α) × EstimatedRTT + α × SampleRTT where α = 0.125. This is an exponentially weighted moving average (EWMA) — gives more weight to recent samples while smoothing outliers. The timeout interval is set as: TimeoutInterval = EstimatedRTT + 4 × DevRTT.

MSS, MTU, and Segment Sizing

MTU (Maximum Transmission Unit): Largest frame the link layer can carry. Typically 1500 bytes for Ethernet.
MSS (Maximum Segment Size): Largest chunk of application data TCP puts in one segment. MSS = MTU − IP header (20) − TCP header (20-60). In the trace: 1500 − 20 − 32 = 1448 bytes.

Delayed ACKs

Instead of ACKing every segment immediately, the receiver waits up to 500ms (or until a second segment arrives) before sending an ACK. This reduces ACK traffic. When the second segment arrives before the timer, the receiver ACKs both at once (cumulative ACK covering 2 segments = 2896 bytes in the trace).

SACK (Selective Acknowledgment)

Negotiated during the handshake via TCP options. Without SACK, a cumulative ACK can only say "I have everything up to byte X." With SACK, the receiver includes additional SACK blocks listing non-contiguous byte ranges it has received — letting the sender retransmit only the specific missing segments instead of everything after the gap.

Q1 How does the SYN-ACK's acknowledgment number relate to the SYN's sequence number? ▼
The SYN-ACK's ack number = client's ISN + 1. The SYN segment carries no data, but it "consumes" one sequence number (so the next expected byte from the client is ISN + 1). In the trace: client SYN seq=4236649187, server SYN-ACK ack=4236649188. This confirms to the client that the server received its SYN.
Q2 How did you identify retransmissions? Were there any? ▼
I checked three ways: (1) Look for duplicate sequence numbers — a retransmitted segment has the same seq as a previously sent one. (2) Wireshark automatically labels retransmissions with [TCP Retransmission] or [TCP Fast Retransmission] in the Info column. (3) Use the display filter tcp.analysis.retransmission. Result: no retransmissions in this trace — the network was clean.
Q3 Explain what the Stevens plot shows and what phase TCP was in. ▼
The Stevens plot (Time-Sequence Graph) plots sequence numbers vs time. Segments appear as "fleets" — groups of segments sent together. The fleet sizes doubled each RTT: ~1→4→8→16. This exponential doubling is the hallmark of slow start. The fleet spacing of ~28ms matches the RTT. The entire 152KB transfer completed during slow start — the file was small enough that TCP never transitioned to congestion avoidance (linear growth).
Q4 What is the receiver window and did it limit the sender? ▼
The receiver window (rwnd) was ~131,712 bytes (after applying the Window Scale factor negotiated in the handshake). It did not throttle the sender. During slow start, the congestion window (cwnd) was much smaller than rwnd. The sender rate was limited by cwnd, not rwnd. The effective sending rate = min(cwnd, rwnd), and cwnd was the bottleneck.
Q5 Calculate the throughput and explain what determines it. ▼
Throughput = Total data / Transfer time = 153,425 bytes / 0.126 seconds ≈ 1.22 MB/s ≈ 9.74 Mbps. The throughput is determined by: (1) cwnd growth rate during slow start, (2) RTT (shorter RTT = faster cwnd growth = higher throughput), (3) available bandwidth on the path, (4) packet loss rate (no loss = smooth growth). For short transfers like this, slow start dominance means throughput is primarily limited by RTT and the number of RTTs needed to ramp up.
Q6 What is the difference between flow control and congestion control? ▼
Flow control protects the receiver — the receiver advertises how much buffer space it has (rwnd) so the sender doesn't overflow it. It's an end-to-end mechanism between sender and receiver. Congestion control protects the network — the sender probes for available bandwidth by adjusting cwnd. It responds to network signals (packet loss, delays) to avoid causing congestion. The sender transmits at min(cwnd, rwnd).

Packet Header (11 bytes)

Format: !IIBH = network byte order. Fields: seq_num (uint32, 4B) — position in byte stream. ack_num (uint32, 4B) — next expected byte. flags (uint8, 1B) — SYN=0b0001, ACK=0b0010, FIN=0b0100, SYNACK=0b0011. checksum (uint16, 2B) — 16-bit one's complement for error detection. Data follows the header as variable-length bytes.

Checksum Algorithm

The same one's complement checksum used by real UDP/TCP/IP: treat data as 16-bit words, sum them with end-around carry, then take the bitwise NOT. To pack: zero out checksum field → compute over header+data → repack with real value. To verify: receiver recomputes — mismatch means corruption → discard packet.

Stop-and-Wait (rdt 3.0)

_reliable_send(): Send one packet → wait up to 50ms for correct ACK → if timeout or wrong ACK, retransmit. Max 50 retries before ConnectionError. Corrupt packets (failed checksum) are silently discarded via ValueError from Packet.unpack().

recv(): Wait for packet with seq_num == expected_seq. If correct, burst 3 ACKs (redundancy against ACK loss) and return data. If duplicate/out-of-order, re-ACK but discard the data.

Connection Lifecycle

Open (3-way handshake): Client sends SYN(seq=X) → Server sends SYNACK(seq=Y, ack=X+1) → Client sends ACK(ack=Y+1). Both sides synchronize ISNs.

Data Transfer: Send data chunks (1024 bytes max each), stop-and-wait for each. Seq increments by chunk length.

Close (simplified): Send FIN → wait for ACK → drain for 100ms → close socket.

HTTP/1.0 Implementation

One connection per request (connect → send request → receive response → close). GET serves files from www/ with path traversal protection. POST echoes body or delegates to registered handlers. Status codes: 200 OK, 404 Not Found, 400 Bad Request. Request/response parsing splits on \r\n\r\n and uses Content-Length to know when the body is complete.

Your lab uses stop-and-wait (simplest but lowest utilization). In practice, TCP uses a hybrid of Go-Back-N and Selective Repeat (with SACK).

Why Does the Checksum Zero the Field First?

The checksum covers the entire packet including the checksum field itself. To compute it, you set the checksum field to 0 in the bytes, compute the sum over those bytes, then insert the result. To verify, the receiver does the same process — if the received checksum matches the recomputed one, the packet is intact. Alternatively, summing the entire packet (including the checksum) should yield 0xFFFF if intact.

The Four Types of Network Delay

Processing delay — router examines header, determines next hop. Queuing delay — packet waits in router buffer (variable, depends on traffic). Transmission delay — L/R = packet length / link bandwidth — time to push all bits onto wire. Propagation delay — d/s = distance / signal speed — time for a bit to travel through the medium. Total delay = sum of all four at each hop. RTT ≈ 2 × (sum of delays along the path).

Expected Discussion Questions

Q&A ▼
Q1 Walk me through what happens when a packet is corrupted in your implementation. ▼
(1) _send_raw() flips a random bit via corrupt_packet() (XOR the LSB of a random byte). (2) The corrupted bytes arrive at the receiver. (3) Packet.unpack() extracts the header, zeroes the checksum field, recomputes the checksum over the modified bytes. (4) The recomputed checksum doesn't match the received checksum → ValueError raised. (5) The caller catches the exception silently and continues waiting. (6) The sender's 50ms timeout fires without receiving an ACK. (7) The sender retransmits the exact same packet. (8) On success, the receiver gets the intact packet, verifies checksum, and sends ACK.
Q2 How does your receiver handle duplicate packets? ▼
Each RUDPSocket tracks self.expected_seq. In recv(), when a packet arrives: if pkt.seq_num == expected_seq, it's new data — accept it, update expected_seq, send burst ACK, return data to application. If pkt.seq_num != expected_seq, it's a duplicate or out-of-order — send an ACK (so the sender stops retransmitting) but do not return the data to the application. The application layer never sees duplicates.
Q3 Why does the receiver send 3 ACKs instead of 1? ▼
ACKs have no ACK of their own — if a single ACK is lost, the sender times out and retransmits unnecessarily. By sending 3 copies, the probability that all 3 are lost is much lower (p³ instead of p). This is a pragmatic trick to reduce unnecessary retransmissions at the cost of a small amount of extra bandwidth. In real TCP, cumulative ACKs and the sender's retransmission timer handle this differently.
Q4 Explain the ACK number calculation: ack_num = seq_num + max(len(data), 1). Why max(_, 1)? ▼
For data packets, the ACK number is the next expected sequence number: seq_num + len(data). For control packets like SYN and FIN that carry no data, len(data) = 0 — but these packets still need to "consume" a sequence number (just like real TCP). The max(0, 1) = 1 ensures that SYN/FIN advance the sequence number by 1, so the other side knows the control packet was acknowledged.
Q5 Why does the server create a new RUDPSocket for each connection? ▼
HTTP/1.0 is one-request-per-connection. RUDP maintains state: seq_num, expected_seq, target_addr. Each client interaction needs fresh state — otherwise leftover sequence numbers from the previous connection could cause confusion (old ACKs accepted for new data, etc.). This mirrors how real TCP's accept() returns a new socket with fresh state per client.
Q6 How does your HTTP server know when the complete request has arrived? ▼
_recv_http_request() accumulates chunks from rudp.recv() into a buffer. It keeps reading until: (1) It finds \r\n\r\n — the blank line separating headers from body. (2) It parses the Content-Length header from the headers section. (3) It checks if the buffer has at least header_end + 4 + Content-Length bytes. For GET requests, Content-Length is 0, so it stops right after the headers. For POST, it keeps reading until the full body arrives.
Q7 Why are sequence numbers initialized randomly? ▼
To prevent old (stale) packets from a previous connection being mistakenly accepted by a new connection that happens to use the same ports. If both connections started at seq=0, a delayed packet from connection #1 could arrive during connection #2 with a valid sequence number and be accepted as new data. Random ISNs make this collision extremely unlikely. This mirrors TCP's ISN randomization (which also has a security motivation — preventing sequence prediction attacks).
Q8 What is the path traversal protection in your server and why is it needed? ▼
Without protection, a request like GET /../../../etc/passwd could escape the web root directory and access system files. The code does: file_path = os.path.normpath(os.path.join(webroot, path.lstrip("/"))) then checks file_path.startswith(webroot). normpath resolves ../ sequences, and the startswith check ensures the resolved path stays inside www/. If it escapes, return 404.
Q9 Can your RUDP checksum detect all errors? Why or why not? ▼
No. The 16-bit one's complement checksum can be fooled by specific error patterns. If two bit flips happen in the same bit position of two different 16-bit words (one adds and one subtracts the same amount), they cancel out and the checksum remains valid despite corruption. Checksums provide error detection (not correction), and they're a probabilistic guarantee — most errors are caught, but not all. For stronger guarantees, you'd use CRCs (used by Ethernet at the link layer).
Q10 Compare your RUDP to real TCP — what's missing? ▼
Missing: (1) Sliding window / pipelining — we use stop-and-wait (window=1), TCP uses a window of N. (2) Flow control — no receiver window advertisement. (3) Congestion control — no slow start, AIMD, fast retransmit/recovery. (4) Full duplex — our send/recv is half-duplex. (5) Proper 4-way teardown — we use simplified FIN/ACK. (6) Nagle's algorithm, delayed ACKs, SACK, window scaling, timestamps. The lab intentionally strips TCP down to its essentials to make the mechanisms visible.
